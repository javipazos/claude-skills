# Terraform 0.12 → 0.13

Primary change: explicit provider source addresses via `required_providers`.

## Recommended Tool

```bash
terraform init
terraform 0.13upgrade -yes .
terraform 0.13upgrade -yes ./submodule1    # repeat for each submodule
terraform fmt
```

The `0.13upgrade` command scans modules for provider usage and adds `source` attributes to `required_providers` blocks. It only exists in 0.13.x binaries.

**Run on each module/submodule separately.** The command only processes the directory you point it at. Third-party providers in submodules (e.g., `datadog/datadog` instead of `hashicorp/datadog`) may not be detected if you only run on the root module.

## Breaking Changes

### Provider Source Addresses
- Providers outside the `hashicorp` namespace now require explicit `source` in `required_providers`
- The `0.13upgrade` command detects and adds these automatically for most providers
- Third-party providers may need manual source correction (e.g., `hashicorp/datadog` should be `datadog/datadog`)

### Single required_providers Block
- Only one `required_providers` block allowed per module — consolidate if you have multiple

### Removed Features
- `terraform import -provider` option removed — add `provider` meta-argument to the resource block instead
- `terraform 0.12upgrade` command no longer exists — must complete 0.12 upgrade before moving to 0.13

### Destroy Provisioner Restrictions
- `when = destroy` provisioners can only reference `self`, `count`, and `each`
- For `null_resource`, put needed values in `triggers` and reference via `self.triggers`

### S3 Backend
- Removed arguments: `lock_table`, `skip_get_ec2_platforms`, `skip_requesting_account_id`
- EC2 instance profile credentials now have lower priority than shared config, web identity, and ECS role credentials
- `AWS_METADATA_TIMEOUT` environment variable no longer used

### Output Behavior
- Root module output changes now require `terraform apply` even without resource changes
- Configurations with unstable output values (referencing data sources that change) must be stabilized

### Platform Requirements
- macOS 10.11 El Capitan minimum (last version to support it)
- FreeBSD 11.2+

## Automated Fixes

### Run 0.13upgrade command on root and all submodules
This handles adding `source` to `required_providers` automatically. Scan for submodules first:
```
Grep pattern: source\s*=\s*"\./  in *.tf files
```

### Verify third-party provider namespaces
After running 0.13upgrade, check generated `required_providers` for incorrect namespaces:
```
Grep pattern: source\s*=\s*"hashicorp/  in *.tf files
```
Cross-reference with actual provider registry names. Common corrections:
- `hashicorp/datadog` → `datadog/datadog`
- `hashicorp/pagerduty` → `pagerduty/pagerduty`
- `hashicorp/cloudflare` → `cloudflare/cloudflare`
- `hashicorp/mongodbatlas` → `mongodb/mongodbatlas`

### Consolidate multiple required_providers blocks
```
Grep pattern: required_providers  in *.tf files
```
If multiple blocks exist in the same module, merge them into one.

### Replace terraform import -provider usage in scripts
```
Grep pattern: terraform import.*-provider  in *.sh, Makefile, *.yml, *.yaml files
```
Remove `-provider` flag and add `provider` meta-argument to the target resource.

### Fix destroy provisioner references
```
Grep pattern: when\s*=\s*destroy  in *.tf files
```
Check provisioner blocks for references beyond `self`, `count`, `each`. Move values to `triggers` in `null_resource`.

### Remove deprecated S3 backend arguments
```
Grep pattern: lock_table|skip_get_ec2_platforms|skip_requesting_account_id  in *.tf files
```
- `lock_table` → `dynamodb_table`

## Manual Steps

### Verify provider source addresses
The upgrade tool may not detect all providers correctly, especially custom, private registry, or third-party providers. Review the generated `required_providers` block in every module.

### Fix unstable outputs
If `terraform plan` keeps showing output changes with no resource changes, the outputs likely reference unstable data sources. The developer must decide how to stabilize them.

## Verification

```bash
terraform fmt -check
terraform validate
terraform plan
```
