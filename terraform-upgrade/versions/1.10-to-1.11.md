# Terraform 1.10 → 1.11

S3 DynamoDB locking deprecated, Azure backend authentication updated.

## Breaking Changes

### S3 Backend — DynamoDB Locking Deprecated
- DynamoDB-related arguments for state locking are deprecated in favor of `use_lockfile`
- DynamoDB can still be used alongside S3-native locking temporarily
- Plan migration to the new locking mechanism

### Test Override Behavior
- Default for overridden values changed to `override_during = apply`
- May affect test files that relied on overrides applying during plan phase

### Azure Backend Authentication
New authentication properties added:
- `use_cli`
- `use_aks_workload_identity`
- `client_id_file_path`
- `client_certificate`
- `client_secret_file_path`

No existing properties removed, but review if using Azure backend.

## Automated Fixes

### Check for DynamoDB locking arguments in S3 backend
```
Grep pattern: dynamodb_table  in *.tf files
```
If found in an S3 backend block, flag as deprecated. The developer should plan migration to `use_lockfile = true`.

### Check test override behavior
```
Grep pattern: override_during  in *.tftest.hcl files
```
If test files rely on specific override timing without explicit `override_during`, they may behave differently.

## Manual Steps

### DynamoDB to S3 native locking migration
This is not urgent but should be planned. The migration involves:
1. Adding `use_lockfile = true` to the S3 backend
2. Testing that state locking works with the new mechanism
3. Eventually removing the `dynamodb_table` argument

### Azure backend users
Review the new authentication options if using the Azure backend. No breaking changes, but new capabilities available.
