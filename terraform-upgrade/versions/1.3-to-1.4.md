# Terraform 1.3 → 1.4

Relatively minor changes. Character encoding adjustments and provider cache behavior change.

## Breaking Changes

### Character Encoding
- `textencodebase64` with `"GB18030"` now encodes € as `0xA2,0xE3` per standard
- `textencodebase64` with `"GBK"` / `"CP936"` now encodes € as `0x80` matching Windows API

### IDNA2008 Hostname Resolution
- Terraform now uses non-transitional IDNA2008 mapping during `terraform init`
- May cause different punycode hostnames for non-ASCII domain names in module sources or provider registries

### Provider Cache Behavior
- `terraform init` ignores global provider cache entries unless checksums are already in the dependency lock file
- First `terraform init` in a new workspace may re-download providers even if they exist in the global cache

### Plan Renderer
- Completely rewritten for future cloud integration
- Output should be materially the same as 1.3 but formatting may differ slightly

## Automated Fixes

No automated code fixes typically needed. These changes affect runtime behavior rather than configuration syntax.

### Check for textencodebase64 with GB18030/GBK
```
Grep pattern: textencodebase64.*GB18030|textencodebase64.*GBK|textencodebase64.*CP936  in *.tf files
```
If found, flag for developer review — the euro symbol encoding changed.

### Check for non-ASCII module sources
```
Grep pattern: source\s*=  in *.tf files
```
If any module sources contain non-ASCII characters (internationalized domain names), flag for review.

## Manual Steps

### Provider cache users
If relying on the global provider cache (via `plugin_cache_dir` in CLI config), the first `terraform init` after upgrading may take longer as it re-downloads providers. This is a one-time cost.

## Verification

```bash
terraform validate
terraform plan
```
