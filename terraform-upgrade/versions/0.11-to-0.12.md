# Terraform 0.11 → 0.12

The most complex upgrade in Terraform's history. Terraform 0.12 introduced HCL2 — a complete overhaul of the configuration language with first-class expressions, proper type system, and stricter validation.

## Pre-Upgrade: While Still on 0.11.14

**Before switching to the 0.12 binary**, run these checks on 0.11.14 (the latest 0.11.x):

```bash
terraform plan               # MUST show no pending changes — do not upgrade with drift
terraform 0.12checklist      # reports syntax issues to fix before upgrading
```

If on an older 0.11.x, install 0.11.14 first to get the checklist command. The checklist is a dry-run of what `0.12upgrade` will do — it reads everything and changes nothing.

Fix everything the checklist reports **while still on 0.11**, before proceeding.

### Pre-Init Blockers — Fix BEFORE Switching to 0.12

These HCL1 patterns prevent `terraform init` from running on 0.12, and the upgrade tool cannot fix them because it never gets to run:

#### Quoted map keys with dots inside blocks
HCL1 allowed quoted keys with dots as block attributes. HCL2 does not.
```
Grep pattern: "\w+\.\w+"  inside resource/data blocks in *.tf files
```
```hcl
# Before — blocks init on 0.12
advanced_options {
  "rest.action.multi.allow_explicit_index" = "true"
}

# After — change block to map assignment
advanced_options = {
  "rest.action.multi.allow_explicit_index" = "true"
}
```

#### required_providers block present in config
The 0.12upgrade tool uses the HCL1 parser which only knows `required_version` and `backend` inside the `terraform {}` block.
```
Grep pattern: required_providers  in *.tf files
```
If found, **remove the entire `required_providers` block** before running the 0.12 upgrade. Re-add it at the 0.13 step. Keep version constraints in provider blocks (0.11 style) for now.

#### Mixed-version syntax
Files may contain syntax from multiple Terraform versions (e.g., `required_version = "0.11.15"` alongside 0.13+ `source` syntax in `required_providers`). No terraform version can parse these files.
```
Grep pattern: required_version  in *.tf files
```
Verify the file is valid HCL1. Remove any 0.13+ syntax. Simplify to what 0.11/0.12 can parse.

## Recommended Tool

```bash
terraform init
terraform 0.12upgrade -yes .
terraform 0.12upgrade -yes ./submodule1    # repeat for each submodule
terraform fmt
```

The `0.12upgrade` command is the primary migration tool. It rewrites HCL syntax automatically and adds `TF-UPGRADE-TODO` comments where manual review is needed. It only exists in 0.12.x binaries.

**Run on each module/submodule separately** — the command only processes the directory you point it at.

If the tool refuses with "Module already upgraded" because `required_version` looks like 0.12+, use:
```bash
terraform 0.12upgrade -force -yes .
```

## Breaking Changes

### Expression Syntax (HCL2)
- Expressions can now be used directly as values: `ami = var.ami` instead of `ami = "${var.ami}"`
- String interpolation-only expressions (`"${var.foo}"`) still work but are flagged by `terraform fmt`
- First-class expressions for lists, maps, conditionals, and loops

### Type System Now Strict
- HCL1 silently coerced nested structures to flat maps. HCL2 enforces types strictly.
- A module receiving `{ type = "map", default = { key = "val" } }` as a `map(string)` will fail — the nested structure doesn't match
- Data structures must match their declared type exactly

### file() Rejects Non-UTF8 Content
- In 0.11, `file()` was lenient with encoding
- In 0.12+, `file()` requires valid UTF-8
- Binary files must use `filebase64()` instead
```
Grep pattern: base64encode\(file\(  in *.tf files
```
Replace `base64encode(file(...))` with `filebase64(...)`.

### Variable Validation
- All variables passed via `-var` and `-var-file` must match declared variables — undeclared variables now cause errors
- Automation that passes a fixed set of variables must switch to `TF_VAR_` environment variables (silently ignored if undeclared)

### Provider Plugin Protocol
- Wire protocol changed completely — ALL providers built for 0.11 are incompatible
- Must use providers built for 0.12+ (run `terraform init` to download compatible versions)
- Terraform 0.12 only supports **plugin protocol 5**. Modern provider versions use protocol 6 and will fail with:
  `Incompatible API version with plugin. Plugin version: 6, Client versions: [5]`

**The agent must pin providers to protocol-5-compatible versions BEFORE running `terraform init`.** Without pinning, init downloads the latest versions which are likely incompatible.

Add version constraints in provider blocks (0.11 style, since `required_providers` is removed for this step):

```hcl
provider "aws" {
  version = "~> 3.0"
  region  = var.region
}
```

Common providers and their last protocol-5-compatible major versions:

| Provider | Pin to | Why |
|----------|--------|-----|
| aws | `~> 3.0` | v4+ uses protocol 6. v3 also has smaller binary (helps Rosetta 2) |
| google | `~> 4.48.0` | v5+ uses protocol 6 |
| azurerm | `~> 2.0` | v3+ uses protocol 6 |
| kubernetes | `~> 2.0` | v3+ uses protocol 6 |
| helm | `~> 2.0` | v3+ uses protocol 6 |
| datadog | `~> 2.0` | v3+ uses protocol 6 |
| cloudflare | `~> 3.0` | v4+ uses protocol 6 |
| null | `~> 3.0` | compatible across versions |
| random | `~> 3.0` | compatible across versions |
| template | `~> 2.0` | deprecated but still works on 0.12 |
| local | `~> 2.0` | compatible across versions |

For providers not listed, check the provider's changelog for when it moved to protocol 6 (usually coincides with dropping Terraform 0.12 support). When in doubt, pin to a major version that was current during 0.12's lifetime (2019-2020).

**Document every pin** — all version constraints must be restored to desired versions at the final migration step.

### Large Providers Under Rosetta 2
- On ARM64 with Rosetta 2, very large provider binaries (e.g., AWS v5 at ~700MB) can timeout when the upgrade tool tries to start the plugin
- Error: "failed to load provider: timeout while waiting for plugin to start"
- Fix: temporarily pin to an older, smaller provider version for the upgrade step

### State and Plan Format
- State snapshot and saved plan serialization changed completely
- External tools parsing `.tfstate` or plan files must be updated
- Use `terraform show -json` for a more stable interface

### Provider Registry
- Provider downloads moved from `releases.hashicorp.com` to `registry.terraform.io`

### Validate Command
- `terraform validate` has a smaller scope — focuses on syntax and type checking only
- May affect CI/CD pipelines that relied on broader validation

## Automated Fixes

The `terraform 0.12upgrade` command handles most syntax transformations automatically. After running it:

### Search for TF-UPGRADE-TODO comments
```
Grep pattern: TF-UPGRADE-TODO
```
Each TODO requires manual review. Present these to the developer.

### Remove redundant interpolation-only expressions
The upgrade tool converts most, but `terraform fmt` catches remaining `"${var.foo}"` patterns:
```
Grep pattern: "\"\$\{[^}]+\}\""  in *.tf files
```
Replace `"${expr}"` with just `expr` where the entire string is a single interpolation.

### Replace file() for binary content
```
Grep pattern: base64encode\(file\(  in *.tf files
```
Replace `base64encode(file("path"))` with `filebase64("path")`.

## Manual Steps

### Review TF-UPGRADE-TODO comments
The upgrade tool marks ambiguous cases. The developer must review each one and decide the correct type or expression.

### Fix type mismatches from HCL1 coercion
If `terraform validate` or `terraform plan` shows type errors (e.g., "element 'default': string required"), the data structures passed to modules or resources need restructuring to match HCL2's strict type system. The developer must decide the correct structure.

### Update external tooling
Any scripts or CI pipelines that parse `.tfstate` files or plan output directly need updating to use `terraform show -json`.

### Update variable passing in automation
If scripts pass variables via `-var` that may not be declared in all configurations, switch to `TF_VAR_` environment variables.

### Provider compatibility
Verify all providers have 0.12-compatible releases. Custom or internal providers need rebuilding against the new plugin SDK. Remember to restore provider version pins at the final migration step.

terraform fmt -check
