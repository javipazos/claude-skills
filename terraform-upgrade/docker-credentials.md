# Docker Credential Forwarding for Terraform

When running old Terraform versions in Docker (because ARM64 native builds don't exist before v1.0.2), credentials must be forwarded into the container for backend access and provider authentication.

## Base Command Pattern

```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  [CREDENTIAL_MOUNTS] \
  hashicorp/terraform:VERSION \
  COMMAND
```

Replace `VERSION` with the target Terraform version (e.g., `0.12.31`, `0.13.7`, `0.14.11`).

## State Persistence Between Commands

Each `docker run` is stateless. The `.terraform` directory (providers, modules, plugins) must persist between commands. Two approaches:

### Option A: Chain commands in a single run (recommended)
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  [CREDENTIAL_MOUNTS] \
  hashicorp/terraform:0.12.31 \
  sh -c "terraform init && terraform 0.12upgrade -yes . && terraform fmt"
```

### Option B: Mount .terraform as a persistent volume
```bash
-v $(pwd)/.terraform:/workspace/.terraform
```

This preserves downloaded providers and modules across separate `docker run` invocations.

## AWS (S3 Backend)

### Mount credential files (works with profiles, SSO, assume-role)
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -v ~/.aws:/root/.aws:ro \
  -e AWS_PROFILE \
  hashicorp/terraform:0.12.31 \
  init
```

The `:ro` mount is read-only — the container cannot modify your local credentials.

If using a named profile, pass `-e AWS_PROFILE=profile-name` so Terraform picks the right one.

### Pass environment variables (works with static keys, CI/CD)
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -e AWS_ACCESS_KEY_ID \
  -e AWS_SECRET_ACCESS_KEY \
  -e AWS_SESSION_TOKEN \
  -e AWS_DEFAULT_REGION \
  hashicorp/terraform:0.12.31 \
  init
```

Using `-e VAR_NAME` without `=value` forwards the variable from the host environment.

### SSO sessions
SSO tokens live in `~/.aws/sso/cache/`. Mount the entire `.aws` directory to include them:
```bash
-v ~/.aws:/root/.aws:ro
```

If the SSO token has expired, the container cannot refresh it (no browser). Refresh on the host first: `aws sso login --profile profile-name`.

## GCP (GCS Backend)

### Application Default Credentials (gcloud auth)
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -v ~/.config/gcloud:/root/.config/gcloud:ro \
  hashicorp/terraform:0.12.31 \
  init
```

This mounts the gcloud config directory which contains the Application Default Credentials generated by `gcloud auth application-default login`.

### Service account key file
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -v /path/to/sa-key.json:/root/credentials.json:ro \
  -e GOOGLE_APPLICATION_CREDENTIALS=/root/credentials.json \
  hashicorp/terraform:0.12.31 \
  init
```

### Environment variable with inline credentials
```bash
-e GOOGLE_CREDENTIALS='{"type":"service_account",...}'
```

Not recommended — the full JSON key is visible in process listings.

## Azure (AzureRM Backend)

### Service principal (most common in CI/CD)
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -e ARM_CLIENT_ID \
  -e ARM_CLIENT_SECRET \
  -e ARM_TENANT_ID \
  -e ARM_SUBSCRIPTION_ID \
  hashicorp/terraform:0.12.31 \
  init
```

### Storage account access key
```bash
-e ARM_ACCESS_KEY
```

### Azure CLI credentials
```bash
-v ~/.azure:/root/.azure:ro
```

Mount the Azure CLI config directory. The container uses the existing login session.

## Terraform Cloud / Enterprise

### Mount CLI config (contains token)
```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -v ~/.terraform.d:/root/.terraform.d:ro \
  hashicorp/terraform:0.12.31 \
  init
```

### Environment variable
```bash
-e TF_TOKEN_app_terraform_io
```

Replace `app_terraform_io` with your hostname using underscores (e.g., `tfe_company_com` for `tfe.company.com`).

## Consul Backend

```bash
-e CONSUL_HTTP_ADDR \
-e CONSUL_HTTP_TOKEN
```

Or mount the Consul config directory if using TLS:
```bash
-v ~/.consul:/root/.consul:ro
```

## Image Availability

The `hashicorp/terraform` Docker Hub images are available for most versions from 0.12 onward. Check availability:

```bash
docker pull hashicorp/terraform:VERSION
```

If an image is not available for a specific patch version, try the latest patch in that minor series (e.g., `0.12.31` is the latest 0.12.x).

For versions before 0.12, images may not exist. In that case, build a custom image:
```dockerfile
FROM alpine:3.12
ADD https://releases.hashicorp.com/terraform/VERSION/terraform_VERSION_linux_amd64.zip /tmp/
RUN unzip /tmp/terraform_VERSION_linux_amd64.zip -d /usr/local/bin/ && rm /tmp/*.zip
ENTRYPOINT ["terraform"]
```

## Security Notes

- Always mount credentials as **read-only** (`:ro`) to prevent the container from modifying them
- Prefer environment variables over mounted files in CI/CD to avoid persisting secrets on disk
- Never bake credentials into Docker images
- After migration is complete, verify no credentials leaked into Terraform state or plan files
